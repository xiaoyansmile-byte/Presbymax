# 下一阶段工作计划

## 当前基线

当前项目已经完成了第一个 Gabor 配对训练项目的功能迁移，以及产品化框架的基础搭建。这个状态可以作为后续开发的重要基线。

### 当前推进状态

- P0 训练闭环和数据一致性：进行中
- 已开始把训练完成后的写回统一到仓库层
- 已明确每日计划的 baseline 规则：首次完成计入计划进度，重复练习仅记录成绩，不再增加完成次数
- 下一步会继续补齐每日计划状态和计划完成次数的一致性策略
- 计划页也已开始使用今日队列作为继续训练入口，不再写死单一训练项目
- 页面统计口径开始拆分为“练习次数”和“计划进度”，避免把复练和 baseline 完成混为一谈
- P1 其他训练项目迁移：已开始，`optictrain-visual-neuro-navigation-demo` 已挂载到框架内并接入今日队列与计划模板；当前已补齐全屏进入、规则弹窗、暂停/继续和结果回传桥，后续继续优化页面内容和交互细节
- `optictrain-navigation` 已进入第二轮交互整理：规则弹窗已合并为单层说明，顶部计时牌与暂停按钮已收拢为单一控制条，左上统计卡改为独立布局，反馈提示位置也在持续修正中，场景图标也开始向更拟物化的 3D 矢量风格调整，Gabor 方向提示也在统一到更一致的箭头 badge 风格；同时补上了计时状态的桥接回传，避免外层倒计时冻结
- `optictrain-navigation` 的驾驶页素材替换已完成并从待办中移除；当前版本已接入用户提供的场景 PNG，并恢复了远景树木/建筑、层级遮挡和车身反馈动效
- `tunnel-symbol-speed.html` 已开始迁移为产品页 `/train/tunnel`：已按“直接点击缺口方向空白区”的交互方式重做，完成/错误反馈与训练结果弹窗也已接入；时长、隧道行进速度、字符移动速度暂以 mock 配置实现
- 今天补齐了训练闭环的日口径重算策略，`gabor-match` 和 `optictrain-navigation` 的今日队列现在会按当天训练记录重新计算，不再依赖旧的静态完成态
- 今天还把 `Navigation` 训练完成后的保存与返回时序收紧，避免结果写回未落稳时首页队列未即时刷新
- `BUG_TRACKER.md` 已整理并验证训练闭环相关问题，`BUG-003` 和 `BUG-005` 目前已从待处理列表中移出
- P2 分析和报告框架：已启动，开始接入时间范围快照、趋势数据和导出入口
- 报告和洞察页已开始共享趋势图、范围切换和最近记录组件，减少重复实现
- 首页统计也已改为直接读取报告快照汇总，和报告页共用同一统计口径
- 导出审计、权限留痕和更细的安全加固暂不进入当前主线，统一归入 P5 稳定性与交付准备阶段处理

### 已确认的基线能力

- 账号、计划、训练、洞察、报告、管理等主路由已经建立
- Gabor 配对训练已经从 Demo 迁移到产品化训练页
- 训练配置已具备管理员侧参数化能力
- SQLite / Postgres 双存储路径已经打通
- `dev.sh` / `setup.sh` 已经可用于本地启动与恢复
- 个人中心、计划页、训练洞察页的产品化 UI 已进入可用阶段

## 下一阶段目标

下一阶段不再只做单页优化，而是围绕“计划驱动训练闭环”继续补齐产品能力。

### P0. 训练闭环和数据一致性

这是下一阶段的首要任务。

- 做实“计划驱动”的每日训练执行闭环
- 训练完成后统一更新：
  - `todayTrainings` 状态
  - 当前计划完成次数
  - 训练记录
  - 相关计划状态
- 明确每日计划部分完成后的交互规则：
  - 待完成训练是否继续显示
  - 是否锁定
  - 是否允许跳过
  - 首页、训练页、个人中心是否保持一致
- 建立统一的数据写回顺序，避免训练页、首页、报告页出现口径不一致

### P1. 其他训练项目从 Demo 迁移

Gabor 已经跑通，但产品还不完整。

- 将其余训练项目从 Demo 逐步迁移到产品体系
- 统一训练模块的执行、结束、计分、记录写回
- 统一训练项在计划中的配置表达
- 统一各训练项完成后的总结和状态回写

### P2. 分析和报告升级

把“列表 + 统计”升级成真正可用的产品能力。

- 增加趋势图
- 增加时间筛选：
  - 7 天
  - 30 天
  - 90 天
- 提供报告导出：
  - JSON
  - PDF
- 让报告页同时服务：
  - 用户复盘
  - 管理员查看
  - 后续留档

### P3. 管理员后台继续完善

在训练闭环稳定之后，继续补齐配置和管理能力。

- 训练配置页的目录化和层级结构继续完善
- Gabor 参数继续做细化配置
- 其余训练项目参数逐步抽象到管理员后台
- 计划模板管理继续完善：
  - 一个计划包含哪些训练项目
  - 训练频率
  - 训练周期
  - 启用 / 停用 / 版本
- 用户管理继续补齐筛选、搜索和计划归属信息

### P4. 个人中心和用户画像

- 年龄、性别、手术类型、手术时间继续投入使用
- 个人中心作为用户画像入口
- 后续用于：
  - 计划推荐
  - 报告分层
  - 统计切片

### P5. 稳定性与交付准备

- 继续清理残留 mock 数据
- 统一 SQLite / Postgres 初始化和迁移脚本
- 保持 `dev.sh`、`setup.sh`、README 的一致性
- 形成可回滚的阶段基线
- 预留自动化回归测试入口
- 补齐导出审计、权限留痕、安全加固、备份恢复和交付级检查

## 推荐执行顺序

1. P0 训练闭环和数据一致性
2. P1 其他训练项目迁移
3. P2 分析和报告升级
4. P3 管理员后台完善
5. P4 个人中心和用户画像
6. P5 稳定性与交付准备

## 本阶段交付标准

如果下一阶段推进顺利，理想状态应达到：

- 所有训练项目都有统一执行闭环
- 计划完成状态和训练记录严格一致
- 报告页具备实际可用的数据分析能力
- 管理员可以维护训练计划和训练参数
- 用户画像字段开始参与产品逻辑
- 新增 bug 都能被记录并回归测试
